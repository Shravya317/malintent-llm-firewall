Set up SQLAlchemy + SQLite. Create 3 tables: ThreatLog (Firewall Log), ActionLog (SEL Action Log — separate from ThreatLog by design), Configuration.
PII Scrubbing pipeline: integrate presidio-analyzer into the logging path. Before any prompt payload is written to ThreatLog, run it through the analyser and replace names, emails, phone numbers, Aadhaar/PAN/card numbers with tokens ([EMAIL_REDACTED], [AADHAAR_REDACTED], etc.). Write a unit test proving the raw PII string cannot be read from the stored log.
Log Tokenisation: implement SHA-256 hashing of the prompt payload. The ThreatLog stores the hash (not the raw text), payload length, risk score, decision, attack category, triggered layers, and timestamp. Implement the Privacy Mode flag so the system stores either scrubbed text (Full Logging) or hash-only (Tokenised Logging). Write a test confirming the original prompt cannot be reconstructed from the hash.
Config Encryption: encrypt all values in the Configuration table using Fernet symmetric encryption (cryptography library). Keys loaded from environment variables only — never written to source code or the DB. Write a test confirming the raw DB file contains no plaintext API keys or system context strings.
SEL Skeleton: build sel/tool_access_controller.py (intercepts LLM tool calls, enforces a whitelist of permitted operations) and sel/permission_validator.py (reads session role + requested tool scope, makes a binary permit/deny decision — does not read the prompt). Wire both as FastAPI middleware on the /api/v1/scan/input endpoint.

Ground rules to protect quality
API contract is frozen after Week 1 sync. The exact shape of RiskResult JSON — including the top-3 FAISS matches and per-layer confidence fields — must be agreed on by end of Week 1 and never changed unilaterally. Tushar cannot rename a field; Shravya cannot add a field she hasn't agreed on. This prevents the classic "backend works, frontend breaks" failure.
Mock data first, real data swap. Shravya builds every page to full visual completion on static mock data. When Tushar's API is ready, it is a one-line base-URL change, not a rebuild. This means UI quality is never blocked by backend timing.
PromptGuard training is Week 2's non-negotiable. The DGX A100 server is a finite resource. If the fine-tuning run fails, debug it and rerun — do not move to Week 3 with a below-target model. Every downstream accuracy claim in the paper depends on this model.
The SEL is Tushar's biggest new addition. Each of the 5 SEL modules has a unit test requirement. Do not commit a module without a test proving its core invariant — e.g. the Dynamic Data Masking test must confirm the LLM never receives the raw phone number, only the masked form.
The Configuration page's new tabs need early design alignment. The Privacy Mode toggle and Permission Roles tab involve subtle UX decisions (what happens if an admin toggles Privacy Mode mid-session? what does deleting a role do to existing sessions?). Tushar and Shravya must agree on these edge cases in Week 4 before either builds their side — behaviour undefined at the seam causes bugs that look like frontend issues but are backend logic gaps.
Both of you know the whole system. Tushar should understand what every frontend page shows and why. Shravya should understand what every detection layer does and why the SEL is a separate trust boundary from the input firewall. If a judge asks Shravya how the Permission Validator works, she should be able to answer clearly. If a judge asks Tushar what the Privacy Mode toggle stores, he should answer immediately.
Comparison Mode is your most powerful asset. Shravya owns the UI but both of you should be able to run it and narrate it fluently. Rehearse the 5 preset attacks until the sequence is automatic. The visceral contrast between the blocked left panel and the compliant right panel is what makes this project memorable to judges.
