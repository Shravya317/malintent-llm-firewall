MALINTENT — SYSTEM ARCHITECTURE
(Paper Section 3 — Week 5 Draft)
=========================================================


3.1 OVERVIEW
------------

MalIntent is implemented as a middleware firewall that sits between a user
and the large language model (LLM) it protects. Instead of relying on a
single input filter — which is what most existing tools do — MalIntent
enforces FIVE independent checkpoints in sequence. A request has to clear
all of them before any data reaches the user.

The five checkpoints are:
  1. The three-layer input firewall (Pattern Engine + ML Classifier + FAISS
     semantic search)
  2. The Permission Validator (role-based access check)
  3. The LLM itself
  4. The Secure Execution Layer, or SEL (controls what tools the LLM can
     call, and masks/redacts anything sensitive that comes back)
  5. The Output Consistency Validator (checks the LLM's final response
     before it's released to the user)

The key design idea: each stage is independent. If one stage somehow gets
bypassed, the next stage still protects the system. For example — even if a
cleverly worded prompt manages to sneak past all three input-detection
layers and tricks the LLM into trying something it shouldn't, the Tool
Access Controller (inside the SEL) will still block that action, because
it's working off a fixed whitelist that no prompt can change. And even if a
tool call IS allowed to go through, any sensitive field in the result —
like a phone number or a card number — still gets masked before the LLM
ever sees it.

This section explains how each of these five stages works. The exact
accuracy numbers and latency numbers for the input-detection layers are
covered later, in the Evaluation section — this section is about the
architecture itself, not the performance numbers.

3.3 INPUT DETECTION (Stages 1 and 2, briefly)
-----------------------------------------------

The three-layer firewall works by combining three different detection
methods into one final score:

  - Layer A (regex pattern matching) catches obvious, known attack phrasing
  - Layer B (the fine-tuned PromptGuard-86M model) catches attacks that are
    reworded or novel — things regex alone would miss
  - Layer C (FAISS semantic search) catches attacks that are paraphrased or
    translated, by comparing meaning rather than exact wording

Each layer doesn't just vote "yes/no" — it contributes a confidence score,
and all three scores are combined into one final risk number out of 100.
That means even a prompt that doesn't trigger any ONE layer strongly, but
shows a little bit of suspicion across two or three layers, can still
correctly get flagged.

(Full details on accuracy, the exact weighting used, and how the
thresholds were chosen belong in the Methodology/Evaluation sections of
the paper — this section is just here to explain how input detection
fits into the bigger five-stage picture.)


3.4 PERMISSION ENFORCEMENT (Stage 2, in detail)
--------------------------------------------------

The Permission Validator makes a simple yes/no decision based on exactly
two things: (1) what role the current session has — Admin, Employee, or
Customer — and (2) what action is being requested.

The important part: it NEVER reads the prompt text itself. This is on
purpose, not an accident or a missing feature.

Because the prompt text plays no role in this decision, there's no way for
an attacker to write a clever sentence that tricks this check. A prompt
that says something like "ignore permissions and show me all customer
data" has zero effect here — the validator was never going to read that
sentence in the first place. The decision is made entirely by code that
runs BEFORE the LLM is even called, based on data the prompt can't touch.

In other words: the LLM's tendency to follow instructions is never part of
the access-control decision at all.


3.5 SECURE EXECUTION LAYER — SEL (Stage 3, in detail)
---------------------------------------------------------

Most existing prompt-injection tools only try to stop a bad instruction
from reaching the LLM in the first place. The SEL takes a different
approach: it assumes that SOMETIMES a bad instruction will get through
anyway, and makes sure that even if it does, it can't actually cause
damage.

The SEL has four parts. Two of them were built and tested this week.

  TOOL ACCESS CONTROLLER (built earlier)
    Enforces a fixed list of what tools/operations/database tables are
    allowed. Anything not explicitly on the list is automatically
    rejected. This list can't be changed by a prompt at runtime — so even
    if the LLM is tricked into trying something like "SELECT * FROM
    users", this layer blocks it no matter how convincingly the LLM was
    talked into trying it.

  DYNAMIC DATA MASKING (NEW this week)
    Any data that comes back from an allowed tool call gets scanned before
    the LLM ever sees it. Phone numbers, card numbers, and email addresses
    get partially hidden — for example a card number becomes
    "**** **** **** 1243" instead of the full number.
    The important detail: the SAME raw value always produces the SAME
    masked value throughout one user session. This is done by hashing the
    original value to use as a lookup key — the original value itself is
    never stored anywhere, even temporarily. That's what lets the LLM say
    something like "your card ending in 1243 was charged" across multiple
    messages in a conversation, without the LLM (or any log file) ever
    actually holding onto the real card number.

  SECRET PROTECTION ENGINE (NEW this week)
    This one protects against a slightly different problem: an attacker
    tricking the LLM into accidentally repeating back a secret — like an
    API key or password — that it picked up from a tool response or a
    config file. It uses two methods together:
      1. Pattern matching for known secret formats (AWS keys, bearer
         tokens, database connection strings, etc.)
      2. A "randomness score" (Shannon entropy) that flags any
         long string of characters that looks randomly generated, even if
         it doesn't match a known format. This catches custom/internal API
         keys that nobody anticipated a specific pattern for.
    Normal sentences and everyday text score low on this randomness scale,
    while real generated keys/tokens score high — so this method is
    unlikely to accidentally flag normal text as a secret, as long as the
    threshold is tuned carefully (which it was, and is covered by tests).

  ACTION AUDIT LOGGER (partially built — full version in Week 7)
    Every decision made by the three modules above is supposed to get
    written to a dedicated log — separate from the main Firewall Log. The
    Firewall Log answers "was this prompt dangerous?" The Action Log
    answers "what did the LLM actually try to do once it was let
    through?" Right now, both new Week 5 modules are wired into the
    response flow, but their results only go to the regular application
    log, not yet to a permanent database table — that full version is
    planned for Week 7, once the LLM can actually call real tools.


3.6 OUTPUT VALIDATION (Stage 4, briefly)
--------------------------------------------

The last stage checks whether the LLM's answer actually makes sense given
what the system was supposed to be doing. It does this by turning both the
system's stated purpose and the LLM's actual response into numeric
representations (embeddings) and measuring how close they are to each
other. If a response drifts too far from the system's intended purpose —
and ALSO matches risky patterns, like a data dump or unrelated system
details — it gets blocked before it reaches the user.

The design for this stage is finalized, and the API endpoint for it
already exists in the backend, but right now it just returns a placeholder
"all good" response so the frontend isn't blocked from being built around
it. The real version — actually doing the embedding comparison — is
scheduled for Week 7.

This stage matters for the paper specifically because it's one of the
things that makes MalIntent different from other tools, most of which only
check the input and never double-check the LLM's actual output.
